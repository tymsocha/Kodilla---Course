14.1. Inversion Of Control i Dependency Injection

Inversion Of Control jest podejściem do pisania oprogramowania, które pomaga w realizacji pryncypium pojedynczej odpowiedzialności dla klasy. Jednym ze sposobów jego realizacji jest zastosowanie tak zwanego wzorca Dependency Injection, który poniżej omawiamy. Cały Spring opiera się na takim sposobie pisania programów - zanim więc opowiemy więcej o Springu dowiesz się trochę o Dependency Injection.

Spring jest biblioteką składającą się z wielu modułów, służącą do zarządzania tworzeniem i usuwaniem obiektów w tak zwanym Kontenerze. Kontener można traktować jak "worek z utworzonymi obiektami" - jest to miejsce, do którego Spring "wrzuca" każdy utworzony obiekt i tam nim zarządza. Spring posiada wiele modułów, niektóre z nich to:

  • moduły podstawowe: Core, Context, Beans służące do zarządzania Kontenerem z obiektami. Wykorzystywane są one do tworzenia obiektów oraz uzyskiwania do nich dostępu w trakcie działania naszej aplikacji
  • moduły związane z pisaniem aplikacji webowych: Web, Servlet
  • moduły służące do obsługi baz danych: JDBC, ORM, Transactions
  • moduły służące do integracji pomiędzy różnymi aplikacjami: JMS
  • moduły odpowiedzialne za tak zwane programowanie aspektowe: AOP

oraz wiele innych. Więcej na ten temat możesz znaleźć tutaj.

Framework

Bardzo rozbudowane biblioteki, często składające się z wielu modułów, nazywamy Frameworkami. Frameworki oprócz tego, że dostarczają różnych potrzebnych funkcjonalności, bardzo często "wymuszają" na programiście pewien specyficzny sposób programowania, czy też pisania kodu.

Z jednym, nieco mniejszym Frameworkiem, z jakim już się spotkaliśmy - był to Java Collections Framework.

Podobnie jest ze Springiem - bardzo często będziemy się więc posługiwali nazwą Spring Framework.

Polimorfizm - przypomnienie

Polimorfizm

Polimorfizm (inaczej: wielopostaciowość) to cecha programowania obiektowego, polegająca na tym, że obiekty mogą być postrzegane pod różnymi postaciami - jako obiekty klas, według których je utworzono, ale również jako obiekty ich klas nadrzędnych (w rozumieniu dziedziczenia). Mając klasę ogólną Figura i dziedziczącą z niej klasę Kwadrat, możemy powiedzieć, że obiekt kwadrat jest klasy Kwadrat, ale również możemy powiedzieć, że jest klasy Figura.

Nim przejdziemy dalej, musimy sobie przypomnieć na czym polega polimorfizm. Aby zacząć, przygotuj środowisko - utwórz nowy moduł o nazwie kodilla-spring-intro (sposób tworzenia modułu opisany był w module 6.1), stwórz w katalogu src/main/java pakiet com.kodilla.spring.intro, a w nim klasę SpringIntroRunner zawierającą metodę main(String[] args):

package com.kodilla.spring.intro;

public class SpringIntroRunner {
    public static void main(String[] args) {
            System.out.println("Welcome into the intro to spring");
    }
}

Struktura katalogów i pakietów powinna wyglądać następująco:

Rozpocznijmy od pokazania podstawowej postaci polimorfizmu. Stwórz pakiet com.kodilla.spring.intro.shape, a w nim trzy klasy: Figure, Triangle oraz Circle.

Klasa Figure:

package com.kodilla.spring.intro.shape;

public class Figure {
    public void draw() {
            System.out.println("This is a geometric figure");
    }
}

Klasa Triangle:

package com.kodilla.spring.intro.shape;

public class Triangle extends Figure {
    @Override
    public void draw() {
            System.out.println("This is a triangle");
    }
}

Klasa Circle:

package com.kodilla.spring.intro.shape;

public class Circle extends Figure {
    @Override
    public void draw() {
            System.out.println("This is a circle");
    }
}

Pokazane powyżej klasy tworzą hierarchię dziedziczenia z jedną klasą bazową Figure oraz dwoma klasami dziedziczącymi: Circle oraz Triangle. Wszystkie te klasy implementują metodę draw(), która w naszym przypadku po prostu wyświetla w oknie konsoli nazwę figury. Klasy dziedziczące z Figure nadpisują tę metodę swoją implementacją.

Stwórz teraz (w tym samym pakiecie com.kodilla.spring.intro.shape w katalogu src/main/java) klasę Drawer, która będzie korzystała z utworzonych wcześniej klas:

package com.kodilla.spring.intro.shape;

public class Drawer {
    public void doDrawing() {
            Circle circle = new Circle();
            Triangle triangle = new Triangle();

            circle.draw();
            triangle.draw();
    }
}

Metoda doDrawing() tworzy po jednym obiekcie klas Circle oraz Triangle, a następnie wywołuje ich metody draw().

Dopiszemy jeszcze test sprawdzający czy nasze klasy działają. Utwórz więc w katalogu src/test/java pakiet com.kodilla.spring.intro.shape klasę (zestaw testów) DrawerTestSuite zawierającą jeden test o nazwie testDoDrawing():

W momencie wywoływania metody draw() uruchamiana jest metoda odpowiednia dla typu przypisanego do zmiennej obiektu (a nie metoda z klasy Figure). Aby się o tym przekonać, uruchom ponownie test testDoDrawing():

W takim przypadku mówimy, że adres metody w pamięci komputera wyznaczany jest nie na etapie kompilacji kodu, ale podczas działania programu. Zjawisko to nazywamy "późnym wiązaniem metod".

W naszych rozważaniach o polimorfizmie możemy się posunąć jeszcze o krok dalej - możemy zastąpić klasę bazową Figure interfejsem. Stwórz w katalogu src/main/java w pakiecie com.kodilla.spring.intro.shape interfejs o nazwie Shape:

package com.kodilla.spring.intro.shape;

public interface Shape {
    String draw();
}

Interfejs ten posiada jedną metodę o nazwie draw() - analogicznie jak klasa bazowa Figure. Zmodyfikujmy teraz klasy Triangle oraz Circle tak, aby nie dziedziczyły z klasy Figure, lecz aby implementowały interfejs Shape.

Klasa Triangle:

package com.kodilla.spring.intro.shape;

public class Triangle implements Shape {
    @Override
    public String draw() {
            System.out.println("This is a triangle");
                  return "This is a triangle";
    }
}

Klasa Circle:

package com.kodilla.spring.intro.shape;

public class Circle implements Shape {
    @Override
    public String draw() {
            System.out.println("This is a circle");
                  return "This is a circle";
    }
}

Zmodyfikujemy również klasę Drawer tak, aby korzystała z interfejsu Shape:

package com.kodilla.spring.intro.shape;

public class Drawer {
    public void doDrawing() {
            Shape shape;

            shape = new Circle();
            shape.draw();

            shape = new Triangle();
            shape.draw();
    }
}

Zastąpiliśmy całkowicie klasę bazową Figure interfejsem Shape - teraz wykorzystujemy zmienną shape typu Shape. Jak zapewne wiesz, możemy teraz przypisywać do niej dowolne obiekty klas implementujących interfejs Shape. Sprawdźmy również, czy i ten przykład działa - uruchom ponownie test testDoDrawing():

Jak widzisz, test nadal działa, mimo że przeszliśmy z hierarchii klas na interfejs i implementujące go klasy. Również tu widzimy, że adresy metod wiązane są z konkretnymi metodami dopiero podczas działania programu (ang. runtime).

Dependency Injection (DI) oraz Inversion of Control (IoC)

W dotychczas rozważanym przykładzie, klasa Drawer sama zarządzała swoimi zależnościami (ang. dependencies), czyli sama tworzyła obiekty klas Circle oraz Triangle bez względu na to, czy przypisywane były do zmiennych typów Circle i Triangle czy też do zmiennej typu Figure czy wreszcie do zmiennej typu interfejsowego Shape. Gdybyśmy chcieli teraz dopisać kolejną klasę - np. Square - wówczas musielibyśmy zmodyfikować kod klasy Drawer, aby mogła korzystać z obiektów tej klasy.

Spróbujmy teraz "wyciągnąć" zarządzanie zależnościami (obiektami, od których zależna jest klasa Drawer) "na zewnątrz" klasy Drawer. W ten sposób uzyskamy dobrze wyizolowaną klasę, której nie będzie trzeba modyfikować, gdy pojawią się nowe typy obiektów.

Zmodyfikuj kod klasy Drawer w następujący sposób:

package com.kodilla.spring.intro.shape;

public class Drawer {
    final Shape shape;

    public Drawer(final Shape shape) {
            this.shape = shape;
    }

    public String doDrawing() {
            return shape.draw();
    }
}

Klasę Drawer wyposażyliśmy w pole shape typu Shape, które inicjowane jest w momencie tworzenia obiektu klasy Drawer poprzez atrybut konstruktora klasy.

Ważne

Taką konstrukcję kodu, jak pokazano na powyższym snippecie - polegającą na tym, że klasa nie zarządza sama swoimi zależnościami (tworzeniem obiektów, od których zależy), tylko są one klasie podawane przez klasę zewnętrzną (tu: za pośrednictwem konstruktora Drawer(final Shape shape)) - nazywamy wstrzykiwaniem zależności (czyli po angielsku Dependency Injection - DI) przez konstruktor.

Klasa, do której wstrzykujemy zależność, sama nie zarządza swoimi zależnościami - robi to jakaś inna (zewnętrzna) klasa. Dzięki temu nie musimy modyfikować kodu klasy Drawer w sytuacji, gdy pojawią się klasy kolejnych figur - ważne jest jedynie to, aby implementowały interfejs Shape.

Dzięki wstrzykiwaniu zależności możliwe jest praktyczne zrealizowanie jednego z filarów programowania obiektowego - izolacji klas.

Zmodyfikuj teraz kod testów - test testDoDrawing usuń, a w jego miejsce stwórz dwa nowe testy: testDoDrawingWithCircle oraz testDoDrawingWithTriangle:

package com.kodilla.spring.intro.shape;

import org.junit.Test;

public class DrawerTestSuite {
    @Test
    public void testDoDrawingWithCircle() {
            //Given
            Circle circle = new Circle();
            //When
            Drawer drawer = new Drawer(circle);
            String result = drawer.doDrawing();
            //Then
            Assert.assertEquals("This is a circle", result);
    }

    @Test
    public void testDoDrawingWithTriangle() {
            //Given
            Triangle triangle = new Triangle();
            //When
            Drawer drawer = new Drawer(triangle);
            String result = drawer.doDrawing();
            //Then
            Assert.assertEquals("This is a triangle", result);
    }
}

W kodzie testów tworzymy obiekty triangle oraz circle, które następnie wstrzykujemy przez konstruktor do obiektu drawer. Możesz uruchomić zestaw testów DrawerTestSuite, aby przekonać się, że nasze "nowe podejście" działa :)

Klasa Drawer nie wie o wstrzykiwanych jej obiektach niczego, poza tym, że implementują one interfejs Shape. Nawet jeżeli będą go rozszerzały o własne metody, to klasa Drawer i tak nigdy się o nich nie dowie - zawsze będzie je postrzegała w sposób ograniczony przez specyfikację interfejsu Shape. Z punktu widzenia klasy Drawer, wszystkie wstrzykiwane obiekty są "takie same" - są jakimiś implementacjami interfejsu Shape.

Zaprezentowane powyżej podejście nazywamy wstrzykiwaniem zależności (ang. Dependency Injection) lub inaczej inwersją kontroli (ang. Inversion of Control) co ma związek z tym, że następuje tu "odwrócenie kontroli" nad zależnościami - klasa sama ich nie kontroluje. Robi to "ktoś" na zewnątrz klasy.

DI oraz IoC jako filary Springa

Dependency Injection (DI) lub inaczej Inversion of Control (IoC) mają fundamentalne znaczenie we frameworku Spring.

Tak naprawdę Inversion of Control jest trochę szerszym pojęciem. Dependency Injection jest jednym ze sposobów realizacji IoC. Jeżeli chcesz dowiedzieć się więcej - kliknij tutaj. Często jednak pojęcia te używane są zamiennie.

Co to jest Bean oraz POJO

Nim przejdziemy do frameworka Spring musisz jeszcze poznać pojęcie ziarna kodu (ang. bean). Najpierw parę słów o klasach Triangle i Circle, które przed chwilą tworzyliśmy - są to zwykłe, proste klasy Javy, dlatego też często określa się takie klasy skrótem POJO, czyli Plain Old Java Object (pol. zwykły, stary obiekt Javy).

W początkowym okresie prac nad frameworkami zajmującymi się Dependency Injection, wprowadzono specjalny wzorzec klasy, na których działały te frameworki - tak zwany wzorzec Bean. W pierwotnych założeniach klasa napisana według wzorca Bean musiała posiadać:

  • bezparametrowy konstruktor
  • zestaw getterów
  • zestaw setterów

Pierwowzór frameworka Spring - framework EJB (w starszych wersjach) wymagał takich właśnie obiektów. Podkreślało się w nim różnice pomiędzy obiektami typu Bean oraz obiektami typu POJO (nazwa POJO została wprowadzona właśnie dla odróżnienia klas niespełniających wymogów wzorca Bean).

Obecnie czasy się zmieniły i praktycznie każda klasa Javy może być obsługiwana przez frameworki oparte o Dependency Injection (zarówno Spring, jak i nowe wersje EJB - znane częściej pod nazwą JEE, a także inne frameworki). Pojęcie Bean zdewaluowało się i praktycznie Beanem może być dowolne POJO. Termin Bean jednak pozostał w użyciu, dlatego trzeba go wyjaśnić.

Bean

Bean to dowolny obiekt, którego cyklem życia (tworzeniem, wstrzykiwaniem zależności, wstrzykiwaniem jego jako zależność itd.) zarządza framework bazujący na Dependency Injection, w szczególności framework Spring.

Czyli mówiąc prosto - jeżeli Spring utworzy obiekt w oparciu o nasze klasy, to przez sam ten fakt staje się on Beanem :)

Kontener beanów, czyli IoC Container

Centralnym miejscem aplikacji napisanych przy pomocy Spring jest tak zwany Kontener Beanów (inaczej: IoC Container). Kontener jest miejscem, w którym Spring tworzy Beany, czyli obiekty naszych klas. W Springu nie tworzymy obiektów samodzielnie - zajmuje się tym sam Spring, dbając przy okazji o to, aby właściwie poustawiać zależności pomiędzy obiektami (czyli dokonuje wstrzykiwania zależności). Wszystkie obiekty powołane do życia przez Spring przechowywane są w Kontenerze.

Ty, podczas pisania swojego programu, "mówisz" Springowi jakie obiekty (czyli Beany) ma utworzyć w Kontenerze, a potem chcąc ich użyć w swoim programie - pobierasz z Kontenera referencje do nich (czyli "przypisujesz" je do swoich zmiennych) i możesz normalnie wywoływać ich metody tak, jakby obiekty te były utworzone po prostu przez Ciebie.

Programy napisane z użyciem Spring dzielą się więc na dwie części - zestaw obiektów w Kontenerze oraz Twoje lokalne obiekty, które czasem będziesz tworzył poza Kontenerem - te nie będą widoczne dla Springa. Oprócz tego oczywiście klasy, czyli definicje obiektów (Twoich lokalnych, jak i Beanów).

14.2. Konfigurowanie Beanów

Zadajesz sobie zapewne pytanie: "No dobrze, ale po co mi są właściwie potrzebne te Beany oraz Kontener?". Trudno jest w dwóch zdaniach odpowiedzieć na takie pytanie. Wszystkie obiekty składające się na "szkielet" Twojego systemu, które są zarządzane przez Spring wewnątrz kontenera, są Beanami. Takie "scentralizowane" podejście pozwala na skoncentrowaniu się na wywoływaniu metod udostępnianych przez Beany, zamiast na tworzeniu i utrzymywaniu tych obiektów "przy życiu". Dzięki takiemu podejściu (nazywanym swego czasu programowaniem "korporacyjnym") możliwe jest tworzenie dużych aplikacji o złożonej strukturze zależności pomiędzy obiektami w ich wnętrzu. Takie podejście upraszcza proces tworzenia złożonych aplikacji.

W tym submodule poznamy kolejne sposoby tworzenia Beanów. Dowiemy się również, w jaki sposób można skonfigurować istniejące między nimi zależności.

Adnotacja @Bean

W poprzednim przykładzie Spring sam utworzył Beany na podstawie klas oznaczonych adnotacjami @Component i umieścił je w Kontenerze. Istnieje również inny sposób tworzenia Beanów - poprzez oznaczenie adnotacją @Bean metody tworzącej obiekt, który ma się stać Beanem. Zwróć uwagę na różnicę - adnotacji @Component używamy dla klas, a adnotacji @Bean używamy dla metod.

Aby sprawdzić to w działaniu, utwórz w katalogu src/main/java w pakiecie com.kodilla.spring.shape klasę Square, tym razem nie oznaczaj jej jednak adnotacją @Component:

package com.kodilla.spring.shape;

public class Square implements Shape {
    @Override
        public String getShapeName() {
            return "This is a square.";
    }
}

Aby powołać do życia Beana obiektu klasy Square, musimy napisać klasę posiadającą metodę tworzącą obiekty typu Square (tak zwaną fabrykę Beanów). Utwórz w katalogu src/main/java w pakiecie com.kodilla.spring.shape klasę ShapesFactory:

package com.kodilla.spring.shape;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class ShapesFactory {
    @Bean
    public Square createSquare() {
            return new Square();
    }
}

Klasa ShapesFactory posiada w tej chwili jedną metodę createSquare(), która tworzy obiekt klasy Square i zwraca go jako wynik działania metody. Metoda ta została oznaczona adnotacją @Bean, co oznacza, że obiekty "produkowane" przez tę metodę mają zostać przez Springa załadowane do Kontenera i mają stać się Beanami.

Aby całość zadziałała, klasa, w której tworzone są Beany przy pomocy adnotacji @Bean, musi być oznaczona adnotacją @Configuration.

Przypomnienie

Adnotacja @Component służy do oznaczania klas, przy pomocy których Spring ma utworzyć Beany.

Adnotacja @Bean służy do oznaczania metod produkujących obiekty, które mają stać się Beanami.

Aby adnotacje @Bean zadziałały, klasa, która zawiera metody tworzące Beany, musi być dodatkowo oznaczona adnotacją @Configuration.

Powyższe sposoby mogą być stosowane alternatywnie. Niektórych zadań nie da się jednak zrealizować przy pomocy wygodniejszej adnotacji @Component. Przykład już za chwilę :)

package com.kodilla.spring.shape;

import org.springframework.context.annotation.Configuration;
import java.util.Random;

@Configuration
public class ShapesFactory {
    @Bean

            return new Square();

            Shape theShape;
            Random generator = new Random();
            int chosen = generator.nextInt(3);
            if (chosen == 0) {

            } else if (chosen == 1) {
                    theShape = new Circle();
            } else {
                    theShape = new Square();
            }
            return theShape;

Każdy Bean posiada nazwę:

  • W przypadku Beanów tworzonych przy pomocy adnotacji @Component, domyślną nazwą Beana jest nazwa klasy (ale pisana małą literą), np: "circle".
  • W przypadku Beanów tworzonych przy pomocy adnotacji @Bean, domyślną nazwą Beana jest nazwa metody tworzącej Beana, np. "chosenShape".

Metoda getBean(Class class) ma też inną postać: getBean(String name). Umożliwia ona wyszukiwanie Beanów w Kontenerze na podstawie nazwy Beana (a nie jego klasy).

Inne rodzaje komponentów

W Springu możesz się spotkać również z innymi adnotacjami tworzącymi kompomenty (Beany):

  • @Component - podstawowa adnotacja oznaczająca klasę, której obiekt ma się stać Beanem . Najczęściej jest używana do prostych klas, które nie posiadają zależności do innych klas.
  • @Service - adnotacja o działaniu takim samym, jak @Component, stosujemy ją jednak do Beanów, które udostępniają jakieś bardziej złożone usługi.
  • @Controller - adnotacja używana najczęściej podczas pisania tzw. kontrolerów dla aplikacji www. Praktycznie tożsama z adnotacją @Component, jednak warto ją stosować przy kontrolerach usług WWW, o których dowiemy się niebawem.
  • @Repository - adnotacja używana do oznaczania Beanów odpowiedzialnych za przechowywanie danych w trwałych zbiorach danych (np. w bazie danych).

Zależności pomiędzy nimi prezentuje poniższa grafika:

@Component jest ogólnym komponentem reprezentującym każdego Beana. @Service, @Controller, @Repository są bardziej szczegółowymi i wyspecjalizowanymi rodzajami Beanów.

Póki co - wszystkie powyższe adnotacje działają tak samo i można ich używać zamiennie. Nie jest jednak wykluczone, że w kolejnych wersjach Springa zostaną im nadane jakieś specjalne zachowania związane z ich przeznaczeniem. Aby dowiedzieć się więcej na ten temat możesz sprawdzić ten link.

Zasięg beanów

Pojęcie zasięgu Beana opisuje sposób tworzenia i widoczności Beanów w poszczególnych miejscach programu. Każdy Bean może mieć jeden z następujących zasięgów:

singleton Domyślny zasięg Beanów. Oznacza, że w Kontenerze tworzony jest tylko jeden Bean danego typu.
prototype Zasięg pozwalający na utworzenie wielu instancji Beanów danego typu w Kontenerze.
request, session, globalSession, application, websocket Zasięgi wykorzystywane przy programowaniu aplikacji www - na razie nie będziemy ich używali i omówimy je później, podczas tworzenia części WWW naszej aplikacji.

Zasięg Beana możemy ustawić przy pomocy adnotacji @Scope(<scope>), gdzie za <scope> należy wstawić odpowiednio: "singleton", "prototype" itd. Adnotacji @Scope używamy łącznie z adnotacjami @Component (i pochodnymi, jak na przykład @Service, @Controller itd.) oraz @Bean. Na przykład:

package com.kodilla.spring.shape;

import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;

@Component
@Scope("singleton")
public class Circle implements Shape {
    @Override
    public String getShapeName() {
            return "This is a circle.";
    }
}

Zasięg singleton jest zasięgiem domyślnym, więc de facto nie musimy używać adnotacji @Scope, gdy zasięgiem ma być singleton. Zauważ, że przy pisaniu adnotacji @Scope IntelliJ stara się nam pomagać - napisz @Scope("") i ustaw się w środku cudzysłowów, a następnie naciśnij Ctrl+Space - środowisko IntelliJ wyświetli możliwe do użycia w danym momencie opcje (dla aplikacji niewykorzystujących modułu Web Springa będą to tylko zasięgi singleton oraz prototype):

Automatyczna konfiguracja beanów

Zazwyczaj, Beany w pisanych przez nas aplikacjach nie są odizolowanymi od siebie samotnymi wyspami, jak to miało miejsce w naszych powyższych przykładach :) Istnieją pomiędzy nimi zależności - odwołują się wzajemnie do siebie, korzystają z metod innych Beanów itd. Aby mogło tak być, musimy w jakiś sposób powiedzieć Springowi, jakie to są zależności - ich opis nazywamy konfiguracją Beanów.

Konfigurację Beanów możemy przygotować na jeden z trzech sposobów:

  • Konfiguracja przy pomocy adnotacji - to konfiguracja bazująca na adnotacjach @Component oraz @Autowired, którą za chwilę poznamy
  • Konfiguracja przy pomocy kodu Java - to konfiguracja oparta o klasy konfiguracyjne oznaczone adnotacją @Configuration, zawierające metody produkujące Beany
  • Konfiguracja przy pomocy plików xml - nie będziemy jej omawiali w tym kursie, ponieważ jest zanikającym sposobem konfigurowania Beanów

Aby przetestować jak działają różne rodzaje konfiguracji, utwórz w katalogu src/main/java pakiet com.kodilla.spring.library, a w nim klasę LibraryDbController:

package com.kodilla.spring.library;

import org.springframework.stereotype.Repository;

@Repository
public final class LibraryDbController {
    public void saveData() {
            System.out.println("Saving data to the database.");
    }

    public void loadData() {
            System.out.println("Loading data from the database");
    }
}

Klasa ta jest odpowiedzialna za zapisywanie stanu księgozbioru do bazy danych oraz odczytywanie tego stanu z bazy danych. W celu uproszczenia nie będzie nic w bazie danych zapisywała - po prostu wyświetli komunikat o tym, że dane są zapisywane lub odczytywane :) Bazy danych poznasz naprawdę w jednym z kolejnych modułów, jednak póki co zastosujemy tylko "zaślepkę".

Zauważ, że zamiast adnotacji @Component użyliśmy adnotacji @Repository, co wskazuje na to, że nasza klasa odpowiedzialna jest za obsługę składowania danych w trwałych zbiorach (repozytoriach), takich jak na przykład bazy danych. Oczywiście moglibyśmy użyć w tym miejscu adnotacji @Component i nasze rozwiązanie również by działało. Jednak adnotacja @Repository niesie również pewien przekaz informacyjny dla innych programistów - mówi im, że klasa ta służy do obsługi repozytorium danych.

W następnych krokach utworzymy klasę Library, która będzie wykorzystywała klasę LibraryDbController. Zrobimy to na trzy różne sposoby - z wstrzyknięciem Beana przez konstruktor, przez metodę oraz poprzez pole klasy.

Wstrzykiwanie zależności przez konstruktor

Utwórz teraz w tym samym pakiecie (com.kodilla.string.library) klasę Library:

package com.kodilla.spring.library;

import org.springframework.stereotype.Service;
import java.util.ArrayList;
import java.util.List;

@Service
public final class Library {
    private final List<String> books = new ArrayList<>();
    private final LibraryDbController libraryDbController;

    public Library(final LibraryDbController libraryDbController) {
            this.libraryDbController = libraryDbController;
    }

    public void saveToDb() {
            libraryDbController.saveData();
    }

    public void loadFromDb() {
            libraryDbController.loadData();
    }
}

Klasa ta posiada prywatne pole libraryDbController, które przechowuje referencję do obiektu klasy LibraryDbController. Pole to jest inicjowane poprzez konstruktor klasy.

Zauważ, że klasa Library oznaczona jest adnotacją @Service, co oznacza, że jest złożonym komponentem, realizującym różne usługi biznesowe (tak naprawdę możemy napisać po prostu @Component i rozwiązanie również będzie działało).

Nigdzie w kodzie nie wskazujemy jawnie, że Bean reprezentujący obiekt klasy LibraryDbController ma być wstrzyknięty do konstruktora klasy Library. Spring sam rozpozna taką zależność i podczas tworzenia Beanów w Kontenerze, utworzy najpierw Beana na podstawie klasy LibraryDbController, a następnie wstrzyknie go poprzez konstruktor do klasy Library.

Adnotacja @Autowired

Spróbujmy teraz usunąć słowo final przy deklaracji zmiennej libraryDbController i dodać do klasy Library drugi konstruktor - tym razem bezparametrowy (zaznaczony podświetleniem):

package com.kodilla.spring.library;

import org.springframework.stereotype.Service;
import java.util.ArrayList;
import java.util.List;

14.3. Konfiguracja przy pomocy klasy Java

Dlaczego tworzenie konfiguracji dla Springa opisujących powiązania pomiędzy Beanami jest takie ważne? Podczas pisania rzeczywistych systemów ich złożoność jest często ogromna. Bardzo trudno jest zapanować w takich systemach choćby nad tym w jakiej kolejności należy tworzyć poszczególne, wzajemnie z siebie korzystające obiekty - zwłaszcza w środowisku rozproszonym. Jednym z zadań Springa jest wzięcie na siebie tego zadania. My, poprzez konfigurację, mówimy Springowi jedynie jakie są powiązania pomiędzy Beanami. Resztę załatwia Spring bez naszego udziału.

Teraz przejdziemy do opisu innej metody - konfiguracji przy pomocy klas Javy - z tego sposobu będziemy korzystali, gdy nie chcemy się zdawać na "inteligencję Springa", tylko sami chcemy wprost powiedzieć co z czym należy połączyć zależnościami.

Aby jawnie (nie automatycznie) wskazać Springowi w jaki sposób powinien skonfigurować Beany, możemy do tego użyć tak zwanej klasy konfiguracyjnej. Zanim zaczniemy, usuń z klasy Library wszystkie adnotacje @Autowired. Wprowadź też do niej dwa konstruktory - po to aby Spring "nie potrafił" sam wywnioskować jak wstrzyknąć zależność:

Utworzymy teraz klasę konfigurującą. W katalogu src/main/java w pakiecie com.kodilla.spring.library utwórz klasę o nazwie LibraryConfig:

package com.kodilla.spring.library;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class LibraryConfig {
    @Bean
    public Library library() {
            return new Library(libraryDbController());
    }

    @Bean
    public LibraryDbController libraryDbController() {
            return new LibraryDbController();
    }
}

Klasa ta posiada dwie metody tworzące obiekty. Obie metody oznaczone są adnotacją @Bean. Metoda libraryDbController() tworzy obiekt, który jest wstrzykiwany do obiektu tworzonego przy pomocy metody library(). Zwróć uwagę, że wewnątrz metody library() następuje wywołanie konstruktora new Library(libraryDbController()), gdzie w jawny sposób wskazujemy, że do utworzenia obiektu klasy Library należy użyć obiektu klasy LibraryDbController utworzonego przez metodę libraryDbController() - następuje jawne wstrzyknięcie przez konstruktor (nie automatyczne).

Cała klasa LibraryConfig oznaczona jest adnotacją @Configuration, co informuje Springa o tym, że należy użyć metod produkujących Beany z tej klasy do skonfigurowania zależności pomiędzy Beanami. Adnotacje @ComponentScan, @EnableAutoConfiguration, @SpringBootApplication

Jeżeli używasz adnotacji @Component do oznaczania klas, z których mają powstać Beany, możesz skorzystać również z adnotacji @ComponentScan, która wskazuje "od której" klasy (a właściwie pakietu) Spring ma poszukiwać komponentów. Przeszukiwanie odbywa się "w głąb" drzewa pakietów.

Adnotacja @EnableAutoConfiguration włącza automatyczne konfigurowanie Beanów przez Springa. Można wykluczyć konkretne klasy z automatycznej konfiguracji przy pomocy parametru exclude tej adnotacji.

Spring Boot udostępnia jednak bardzo przydatną adnotację @SpringBootApplication, która powoduje, że włączane są wszystkie powyżej wskazane adnotacje. Wystarczy więc umieścić tylko adnotację @SpringBootApplication przy klasie zawierającej metodę main(String[] args) naszej aplikacji. Nie ma wówczas potrzeby stosowania adnotacji @ComponentScan oraz @EnableAutoConfiguration. W naszej aplikacji skorzystaliśmy właśnie z adnotacji @SpringBootApplication:

package com.kodilla.spring;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class SpringRunner {
    public static void main(String[] args) {
        SpringApplication.run(SpringRunner.class, args);
    }
}

Należy jedynie zwrócić uwagę na to, aby klasa zawierająca metodę main(String[] args) umieszczona była w pakiecie "najwyższego poziomu" naszej aplikacji - dzięki czemu wszystkie podpakiety będą skanowane przez Springa w poszukiwaniu komponentów oraz konfiguracji.

Kontekst aplikacji

Kontekst aplikacji jest obiektem przechowującym informacje o konfiguracji Springa - przy jego pomocy możemy uzyskiwać dostęp do Beanóww Kontenerze. Kontekst udostępnia również szereg innych przydatnych funkcji - pozwala sprawdzić jakie Beany znajdują się w Kontenerze, posiada też metody ponownie inicjujące konfigurację Springa. Zapewne nieco już go kojarzysz - spotkaliśmy się z nim w trakcie pisania testów :)

Obiekt kontekstu aplikacji tworzony jest na podstawie klasy implementującej interfejs ApplicationContext. Istnieje wiele klas implementujących ten interfejs - różnią się one sposobem odczytania informacji o konfiguracji Springa. Wiele z nich dotyczy konfiguracji przy pomocy plików xml, której tu nie omawiamy, jako że wychodzi ona z użycia - zamiast tego, we wszystkich naszych przykładach używamy klasy AnnotationConfigApplicationContext, która oparta jest o adnotacje Javy oraz klasy konfigurujące.

Klasa AnnotationConfigApplicationContext posiada kilka konstruktorów. Dla nas szczególnie istotne są dwa z nich:

  • AnnotationConfigApplicationContext(String... basePackages) - tworzy kontekst aplikacji na podstawie wszystkich odnalezionych konfiguracji automatycznych oraz klas konfigurujących w drzewie pakietów aplikacji, począwszy od basePackages "w dół" hierarchii (można podać kilka pakietów).
  • AnnotationConfigApplicationContext(Class<?>... annotatedClasses) - tworzy kontekst aplikacji na podstawie klasy konfigurującej annotatedClasses (można podać kilka klas).

Aby wyświetlić nazwy Beanów załadowanych do Kontenera możemy skorzystać z metody getBeanDefinitionNames() kontekstu. Zwraca ona tablicę String[] z nazwami Beanów. Przy pomocy metody Arrays.stream(<T>[] array) możemy utworzyć stream i wyświetlić nazwy Beanów. Stwórz w zestawie testów LibraryTestSuite nowy test o nazwie testContext():

                        new AnnotationConfigApplicationContext(LibraryConfig.class);

              //When & Then
    System.out.println("===== Beans list: ==== >>");
        Arrays.stream(context.getBeanDefinitionNames())
                        .forEach(System.out::println);
        System.out.println("<< ===== Beans list ====");

Adnotacje @Qualifier oraz @Conditional

Czasem będziemy mieli do czynienia z sytuacją, w której w Kontenerze znajduje się kilka Beanów danego typu - wówczas aby wskazać, który z nich ma być wstrzyknięty jako zależność, możemy skorzystać z adnotacji @Qualifier. Sprawdźmy jak to zrobić w praktyce na przykładzie.

W katalogu src/main/java utwórz pakiet com.kodilla.spring.reader, a w nim klasę o nazwie Book reprezentującą książkę:

package com.kodilla.spring.reader;

public final class Book {
    final String title;

    public Book(final String title) {
            this.title = title;
    }

    public String getTitle() {
            return title;
    }
}

Klasa ta posiada jedynie pole title typu String inicjowane przez konstruktor. Następnie utwórz klasę Reader reprezentującą czytnik książek:

package com.kodilla.spring.reader;

public final class Reader {
    final Book theBook;

    public Reader(final Book theBook) {
            this.theBook = theBook;
    }

    public void read() {
            System.out.println("Reading: " + theBook.getTitle());
    }

Klasa Reader posiada pole theBook, inicjowane przez konstruktor. Posiada również metodę read(), która wyświetla tytuł książki będącej przedmiotem odczytywania.

Stwórz teraz w tym samym pakiecie klasę konfigurującą Beany o nazwie ReaderConfig:

package com.kodilla.spring.reader;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Scope;

@Configuration
public class ReaderConfig {
    @Autowired
    @Qualifier("book1")
    Book book;

    @Bean
    public Reader getReader() {
            return new Reader(book);
    }

    @Bean(name = "book1")
    @Scope("prototype")
    public Book getBookOne() {
            return new Book("The book number one");
    }

    @Bean(name = "book2")
    @Scope("prototype")
    public Book getBookTwo() {
            return new Book("The book number two");
    }
}

Zauważ, że przy metodach getBookOne() oraz getBookTwo() użyliśmy adnotacji @Bean w nieco innej postaci niż dotychczas - uzupełniliśmy ją o argument name, przy pomocy którego możemy nadać tworzonym przez te metody Beanom własne nazwy (zamiast domyślnych). W tym przykładzie obiekt zwracany przez metodę getBookOne() otrzymał nazwę book1, natomiast obiekt zwracany przez metodę getBookTwo() otrzymał nazwę book2.

Na początku klasy konfigurującej znajduje się pole Book book, które wiązane jest automatycznie (adnotacja @Autowired). W związku z tym, że w klasie konfigurującej są dwa Beany typu Book, Spring nie wiedziałby, który z nich powiązać z właściwością klasy book. Dlatego też konieczne było zastosowanie adnotacji @Qualifier z parametrem będącym nazwą Beana, aby wskazać Springowi, który z obiektów ma wstrzyknąć.

Druga z adnotacji, którą chcemy omówić w tym punkcie to adnotacja @Conditional. Pozwala ona na selektywne tworzenie Beanów w zależności od spełnienia lub niespełnienia warunku.

Aby użyć adnotacji @Conditional, musimy wpierw napisać klasę sprawdzającą warunek, która implementuje wbudowany interfejs Springa o nazwie Condition. Interfejs ten ma jedną metodę - matches(ConditionContext context, AnnotatedTypeMetadata metadata), która zwraca true lub false. Przejdźmy do napisania tej klasy - utwórz w katalogu src/main/java w pakiecie com.kodilla.spring.reader klasę o nazwie IfDayIsOddCondition implementującą interfejs Condition:

package com.kodilla.spring.reader;

import org.springframework.context.annotation.Condition;
import org.springframework.context.annotation.ConditionContext;
import org.springframework.core.type.AnnotatedTypeMetadata;
import java.time.LocalDate;

public class IfDayIsOddCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
            return LocalDate.now().getDayOfMonth() % 2 != 0;
    }
}

Metoda matches(...) tej klasy zwraca true gdy bieżący dzień jest dniem nieparzystym, oraz false gdy jest dniem parzystym.

Następnie zmodyfikuj kod klasy konfigurującej ReaderConfig:

package com.kodilla.spring.reader;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Conditional;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Scope;

@Configuration
public class ReaderConfig {
    @Autowired
    @Qualifier("book1")
    Book book;

    @Bean
    public Reader getReader() {
            return new Reader(book);
    }

    @Bean(name = "book1")
    @Scope("prototype")
    public Book getBookOne() {
            return new Book("The book number one");
    }

    @Bean(name = "book2")
    @Scope("prototype")
    @Conditional(IfDayIsOddCondition.class)
    public Book getBookTwo() {
            return new Book("The book number two");
    }
}

Metoda getBookTwo() została opatrzona adnotacją @Conditional(IfDayIsOddCondition.class), która działa w ten sposób, że Bean o nazwie book2 zostanie utworzony tylko wtedy, gdy metoda matches(...) klasy IfDayIsOddCondition zwróci wartość true.

Aby się przekonać czy to działa, dopisz nowy test o nazwie testConditional() do zestawu testów ReaderTestSuite:

14.4. Spring Web

Najszerzej wykorzystywaną obecnie cechą Springa jest to, że bardzo szybko można z jego użyciem napisać aplikacje, którą mogą być udostępniane użytkownikom jako aplikacje www przez przeglądarkę internetową. Spring MVC, który za chwilę wstępnie omówimy jest jednym z najpopularniejszych (i najwygodniejszych) sposobów na pisanie aplikacji www.

Spring posiada wiele użytecznych bibliotek. Poznamy teraz pierwszą z nich - bibliotekę Web.

Spring Web

Biblioteka Spring Web udostępnia szereg klas i interfejsów, których możemy użyć do tworzenia dynamicznych stron internetowych.

Statyczne i dynamiczne strony WWW

  • statyczna strona internetowa - jest to po prostu zbiór plików HTML oraz CSS. Taką stronę możesz zapisać np. u siebie na dysku i otworzyć przy pomocy dowolnej przeglądarki internetowej. Aby udostępnić ją w internecie potrzebny jest serwer udostępniający statyczne strony internetowe (np. bardzo popularny serwer Apache)
  • dynamiczna strona internetowa to taka strona, która jest "generowana" w momencie wywołania - np. gdy przechodzisz w sklepie internetowym do koszyka zakupów, to jego zawartość jest generowana na podstawie tego co wcześniej wybrałeś do zakupu. Takiej strony nie da się zrealizować jako statyczny HTML. Aby móc generować tego typu strony potrzebny jest serwer aplikacyjny (na przykład Tomcat, na którym uruchomione jest oprogramowanie generujące zawartość stron)

Strony WWW, z którymi masz styczność podczas przeglądania internetu możemy podzielić na statyczne i dynamiczne. Oto niektóre różnice:

To wszystko co jest prezentowane użytkownikowi za pośrednictwem stron WWW nazywamy potocznie "front-endem". Składa się na niego szereg technologii, których nie będziemy teraz szczegółowo omawiać. Niektóre ich elementy poznasz jednak w dalszej części kursu. Najważniejsze z nich to:

  • HTML - język znaczników, używany do tworzenia zawartości stron WWW
  • CSS - arkusze stylów mówiące o tym, w jaki sposób strona HTML ma być wyświetlana

Adresy i porty

Każdy serwer dostępny w sieci internet posiada swój adres IP. Do serwerów uruchomionych na Twoim komputerze można się odwoływać poprzez specjalny adres 127.0.0.1, który oznacza po prostu "ten komputer". Ma on również swoją nazwę symboliczną: localhost, której można używać zamiast pisania adresu IP.

Na jednym komputerze może pracować równocześnie wiele serwerów - aby było to możliwe, zajmują one oddzielne tak zwane porty. Domyślnym portem, na którym pracuje serwer Tomcat jest port o numerze 8080.

Dostęp do konkretnego portu na konkretnym serwerze uzyskujemy następująco:

127.0.0.1:8080

Możemy też w przypadku tego konkretnego adresu napisać po prostu

localhost:8080

co oznacza serwer Tomcat zainstalowany na lokalnym (Twoim) komputerze.

Na powyższym schemacie przedstawiono cykl obiegu informacji we wzorcu MVC. Poniżej opisano co po kolei się dzieje:

  1. Użytkownik za pośrednictwem przeglądarki wysyła żądanie do serwera WWW (request) - na przykład klika w przycisk "Pokaż listę użytkowników"
  2. Kontroler "wie", która z usług biznesowych pobiera z systemu i zwraca listę użytkowników. Wywołuje więc metodę z tej usługi biznesowej.
  3. Usługa biznesowa generuje dane - listę użytkowników (model) i przekazuje ją z powrotem do kontrolera.
  4. Kontroler "wie" dla jakiej przeglądarki ma wygenerować odpowiedź (zwykłej lub np. mobilnej) i wywołuje metodę z odpowiedniego generatora widoku, przekazując mu dane otrzymane z usługi biznesowej.
  5. Generator widoku tworzy stronę w HTML odpowiednią dla przeglądarki użytkownika, zawierającą listę użytkowników i przekazuje ją do kontrolera.
  6. Kontroler odsyła użytkownikowi gotową stronę HTML z oczekiwanymi danymi (response).

W ten sposób, przy użyciu wzorca MVC obsługiwane są żądania napływające z przeglądarek klientów (użytkowników) do serwera tworzącego dynamiczne strony WWW. Dzięki takiemu podejściu po stronie serwera dostępne są klasy z wyizolowaną odpowiedzialnością i nie dopuszczamy do zastosowania złej praktyki, w której "jedna klasa robi wszystko".

Wzorzec Spring MVC

Wzorzec Spring MVC jest nieco rozbudowaną wersją omówionego przed chwilą wzorca MVC. Spójrz na poniższy diagram:

Przeanalizujemy kolejne kroki:

  1. Przeglądarka na komputerze użytkownika wysyła żądanie do serwera www, poprzez wywołanie konkretnego adresu na stronie, na przykład: www.myapp.com/users
  2. Dispatcher (czyli kontroler frontowy) używając informacji o adresie aplikacji jaki został wywołany (tu: "/users") odpytuje się usługi Handler Mapping jakiej usługi biznesowej użyć do obsługi żądania
  3. Handler Mapping zwraca informację o usłudze biznesowej
  4. Dispatcher wywołuje usługę biznesową, która powinna przygotować dane do obsługi żądania
  5. Usługa biznesowa zwraca obiekt Java zawierający żądane dane (np. listę użytkowników) oraz informację o rodzaju widoku, który powinien być zastosowany do ich prezentacji
  6. Dispatcher przesyła informację o rodzaju widoku do View resolvera
  7. View resolver zwraca informację o konkretnym generatorze widoków, który potrafi zwracać widoki wybranego rodzaju
  8. Dispatcher przekazuje do generatora widoków otrzymany obiekt Javy z danymi
  9. Generator widoku tworzy wynikowy plik HTML zawierający oczekiwane dane - przekazuje go z powrotem do Dispatchera
  10. Dispatcher przekazuje wygenerowany plik HTML do przeglądarki użytkownika
