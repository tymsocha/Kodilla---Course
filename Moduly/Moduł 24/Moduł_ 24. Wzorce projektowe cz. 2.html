Przedstawienie problemu na przykładzie

Jako przykład weźmiemy pod uwagę sklep internetowy udostępniający funkcjonalność składania zamówień - możliwą do wywołania bez pośrednictwa GUI (ang. graphic user interface) - w związku z tym, inny system może podłączyć się do sklepu internetowego i złożyć zamówienie za pośrednictwem jakichś udostępnionych przez sklep usług (podobnie działa np. Ceneo).

Nasz sklep internetowy posiada następujące funkcjonalności dostępne dla programistów:

  • Metodę Long openOrder(Long userId) - otwierającą nowe zamówienie dla użytkownika o podanym identyfikatorze (userId); metoda zwraca identyfikator zamówienia; w sytuacji gdy użytkownik nie jest zautoryzowany metoda zwraca -1
  • Metodę void addItem(Long orderId, Long productId, double qty) - dodającą pozycję do zamówienia; do zamówienia o identyfikatorze orderId dodawany jest produkt o identyfikatorze productId w ilości qty (od ang. quantity) sztuk.
  • Metodę boolean removeItem(Long orderId, Long productId) - usuwającą pozycję z produktem o identyfikatorze productId z zamówienia o identyfikatorze orderId; zwraca true gdy pozycja została usunięta oraz false gdy nie było takiej pozycji w zamówieniu
  • Metodę BigDecimal calculateValue(Long orderId) - obliczającą wartość do zapłaty za zamówienie o identyfikatorze orderId
  • Metodę boolean doPayment(Long orderId) - przeprowadzającą proces płatności za zamówienie o identyfikatorze orderId - zwraca true gdy zamówienie zostało (lub było już wcześniej) opłacone oraz false jeżeli nie udało się opłacić zamówienia
  • Metodę boolean verifyOrder(Long orderId) - dokonującą ostatecznej weryfikacji zamówienia (stany magazynowe, status płatności); zwraca true gdy zamówienie jest gotowe do realizacji oraz false jeżeli coś z zamówieniem jest nie tak jak powinno (np. nie jest opłacone)
  • Metodę boolean submitOrder(Long orderId) - zatwierdzającą zamówienie o identyfikatorze orderId do realizacji
  • Metodę void cancelOrder(Long orderId) - anulującą otwarte zamówienie o identyfikatorze orderId

Przypomnienie typu BigDecimal

Typ BigDecimal służy do przechowywania oraz wykonywania operacji na bardzo dużych liczbach.

Nowy obiekt tworzymy w następujący sposób:

BigDecimal myBigNumber = new BigDecimal("12345");

Do dyspozycji mamy również stałe:

  • BigDecimal.ZERO
  • BigDecimal.ONE

Typ BigDecimal udostępnia również kilka przydatnych metod, między innymi:

  • a.subtract(b) - zwraca obiekt BigDecimal będący wynikiem odjęcia liczby b od liczby a (gdzie a i b są obiektami BigDecimal)
  • a.multiply(b) - zwraca obiekt BigDecimal będący wynikiem pomnożenia liczby a przez liczbę b (gdzie a i b są obiektami BigDecimal)
  • a.compareTo(b) - zwraca:
    • wartość > 0 gdy a jest większe od b
    • wartość = 0 gdy a jest równe b
    • wartość < 0 gdy a jest mniejsze od b

Lombok a wzorce projektowe

Ostatnio używaliśmy biblioteki Lombok do generowania konstruktorów, getterów i setterów. W module o wzorcach projektowych postanowiliśmy jednak tego nie robić. Dlaczego? Ponieważ często podczas stosowania wzorców będziemy mieli do czynienia z:

  • prywatnymi (ukrytymi) konstruktorami
  • niestandardowymi getterami i setterami, które mają bardziej złożoną logikę (np. inicjalizacja zmiennej instancyjnej w getterze we wzorcu Singleton)

package com.kodilla.patterns2.facade;

import org.junit.Test;
import org.junit.runner.RunWith;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.junit4.SpringRunner;

import java.math.BigDecimal;

@RunWith(SpringRunner.class)
@SpringBootTest
public class ShopServiceTestSuite {
    @Autowired
    private ShopService shopService;

    @Test
    public void testShopServiceSubmitOrder() {
        long orderId = shopService.openOrder(1L);
        System.out.println("Registering new order, ID: " + orderId);
        if (orderId > 0) {
            shopService.addItem(orderId, 10L, 2);
            System.out.println("Adding item: 10, 2 pcs");
            shopService.addItem(orderId, 216L, 1);
            System.out.println("Adding item: 216, 1 pc");
            shopService.addItem(orderId, 25L, 1);
            System.out.println("Adding item: 25, 1 pc");
            shopService.addItem(orderId, 100L, 2.5);
            System.out.println("Adding item: 100, 2.5 pcs");
            shopService.addItem(orderId, 11L, 3);
            System.out.println("Adding item: 11, 3 pcs");
            if (shopService.removeItem(orderId, 100L)) {
                System.out.println("Item 100 was deleted from order");
            } else {
                throw new IllegalStateException("Order data is corrupted");
            }
            BigDecimal value = shopService.calculateValue(orderId);
            System.out.println("Order value is: " + value + " USD");
            if (shopService.doPayment(orderId)) {
                System.out.println("Payment for order was done");
            } else {
                System.out.println("Payment was rejected");
            }
            if (shopService.verifyOrder(orderId)) {
                System.out.println("Order is ready to submit");
            } else {
                System.out.println("Order cannot be submitted");
            }
            if (shopService.submitOrder(orderId)) {
                System.out.println("Order submitted - have a nice day!");
            } else {
                System.out.println("Order not submitted - cancelling");
                shopService.cancelOrder(orderId);
                System.out.println("Order is cancelled");
            }
        } else {
            System.out.println("Access denied. User is not authenticated.");
        }
    }
}

Rozwiązanie problemu - fasada

Najprostszym i doraźnym (możliwym do natychmiastowego zastosowania) rozwiązaniem opisanych wyżej problemów jest użycie wzorca Fasada. Wzorzec ten nie ma żadnej ściśle zdefiniowanej postaci. Często nazywany jest też po prostu wzorcem API.

Polega on na stworzeniu prostego i przejrzystego API, które stanowi dodatkową warstwę abstrakcji pomiędzy użytkownikiem (czyli innym programistą), a "wnętrzem" naszego kodu. Fasada bierze na siebie ciężar obsługi naszego wewnętrznego "flow" procesu. Fasada powinna również w razie problemów rzucać eleganckie wyjątki, o czytelnych dla programisty opisach.

Przedstawienie problemu

Nasz portal taksówkowy kalkuluje usługi w następujący sposób:

  1. za każdy kurs nalicza opłatę podstawową w wysokości 5 złotych
  2. do powyższej opłaty dolicza wartość kursu wyliczoną według operatora taksówek, do którego skierowano żądanie (Uber, MyTaxi, Taxi itd.)

  4. do powyższych kwot czasem doliczane jest kolejne 2 złote - jeżeli klient wybrał usługę dodatkową Fotelik Dla Dziecka (2 złote za każdy fotelik)

Można by oczywiście wszystkie parametry usługi przekazywać jako argumenty konstruktora klasy - wówczas liczba klas implementujących interfejs byłaby mniejsza. 24.3. Programowanie aspektowe w Spring Boot

Podczas programowania często spotkasz się z sytuacją, w której konieczne będzie wykonanie jakichś operacji "przekrojowo" w całym programie. Na przykład - przed wywołaniem każdej metody dowolnego beana, chcielibyśmy ten fakt zapisać w logach dla celów diagnostycznych (śledzenie przebiegu wykonywania programu). Inny przykład - przy każdej operacji bazodanowej, chcielibyśmy zapisać w logach ile czasu zajęło jej wykonywanie. Takie przekrojowe funkcjonalności (niezwiązane z konkretną metodą czy klasą) nazywamy aspektami (ang. aspect).

Programowanie oparte o użycie aspektów doczekało się wręcz osobnej nazwy - jest to programowanie zorientowane aspektowo (ang. AOP - aspect oriented programming). Istnieje też sporo wzorców projektowych wykorzystujących mechanizm aspektów - na przykład wzorzec Interceptor. My jednak tym razem nie skupimy się na konkretnym wzorcu lecz na całym mechanizmie AOP, ponieważ warto go znać.

Przypomnienie matematyczne - silnia

Silnia jest operacją matematyczną zapisywaną przy pomocy operatora ! (wykrzyknik). Wynikiem n! jest iloczyn liczb naturalnych od 1 do n, czyli:

5! = 1 * 2 * 3 * 4 * 5 = 120

przy czym 0! jest równe 1.

Silnia (ang. factorial) jest operatorem bardzo dobrze nadającym się do zaprezentowania techniki programowania, którą nazywamy rekurencją (ang. recurrency).

Naszą metodę skonstruujemy w oparciu o spostrzeżenie, że:

5! = 4! * 5

i dalej:

4! = 3! * 4

i tak dalej, aż do:

3! = 2! * 3
2! = 1! * 2
1! = 1

Widzimy więc, że aby obliczyć silnię z n, należy n pomnożyć przez (n-1)!, tak długo, aż dotrzemy do obliczenia 1!, które z definicji wynosi 1.

Wartość silni wraz ze wzrostem wartości argumentu bardzo szybko rośnie, dlatego też na potrzeby silni skorzystamy z klasy BigDecimal - już kiedyś z niej korzystaliśmy przy obliczaniu ilości ziaren piasku na świecie :)

Opiszmy to jeszcze raz - załóżmy, że dopóki n jest większe od 1, factorial(n) jest równe n * factorial(n-1). Kiedy n jest równe 1, metoda factorial(1) zwraca 1.

W związku z tym, gdybyśmy chcieli obliczyć factorial z liczby 3, wywołania wyglądałyby mniej więcej tak:

factorial(3) = 3 * factorial(2), gdzie:

factorial(2) = 2 * factorial(1), gdzie:

factorial(1) = 1

Skoro factorial(1) jest równy 1 - zgodnie z tym, co ustaliliśmy przed chwilą, to factorial(2) jest równe 2 (czyli 2 * factorial(1), czyli 2 * 1) - tę wiedzę możemy wykorzystać do obliczenia silni z 3.

Skoro rekurencyjny wzór na silnię to n! = n * (n-1)! (dla n większego niż 1, dla liczby 1 silnia zwraca po prostu 1), a (n-1)! jest w tym przypadku równe 2, to nasze działanie wygląda następująco:

3! = 3 * 2 (czyli 3 * factorial(2)), więc wynikiem jest 6.

Jak widzisz, nasze wywołanie factorial(3) potrzebuje wiedzieć jaki jest wynik factorial(2), aby wykonać swoje obliczenie. factorial(2) z kolei potrzebuje wiedzieć jaki jest wynik factorial(1) - w tym przypadku sytuacja jest prosta - jest to po prostu 1, więc factorial(2) może zwrócić swój wynik na tej podstawie, dzięki czemu factorial(3) również może zwrócić swój wynik.

Na tym polega rekurencja w obliczeniach, takich jak np. obliczanie silni - funkcja wywołuje siebie samą dopóki nie znajdzie prostej, jednoznacznej "odpowiedzi" niebędącej wywołaniem (w naszym przypadku był to wynik factorial(1), czyli po prostu wartość 1) - wtedy "wraca" z tą informacją w górę, aż do pierwszego wywołania.

Mam nadzieję, że jest to dla Ciebie zrozumiałe - nie przejmuj się, rekurencja jest dosyć trudnym zagadnieniem kiedy spotyka się ją po raz pierwszy i wymaga po prostu właściwego "nastawienia" - tak samo jak np. programowanie obiektowe, jednak kiedy już ją zrozumiesz, jest niesamowicie potężnym narzędziem w niektórych przypadkach. To tyle o rekurencji - przejdźmy do testowania kodu :)

Następnie musimy oznaczyć metodę logEvent() jedną z adnotacji definiujących zdarzenie w aplikacji:

  • @Before - zdarzenie występuje każdorazowo przed wywołaniem metody
  • @AfterReturning - zdarzenie występuje po zwróceniu wartości przez metodę w sytuacji gdy zakończyła się bez błędu
  • @AfterThrowing - zdarzenie występujące po tym jak metoda rzuci wyjątkiem
  • @After - występuje zawsze po zakończeniu wykonywania metody bez względu na to czy wykonanie zakończyło się pomyślnie, czy też rzuciło wyjątek
  • @Around - specjalne zdarzenie "otaczające" wywołanie metody - omówimy je dokładniej nieco dalej

Oznaczenie metody logEvent() jedną z powyższych adnotacji spowoduje, że stanie się ona tak zwaną "poradą" (ang. advice), czyli kodem wykonywanym w reakcji na zdarzenie.

Zaczniemy od przykładu z adnotacją @Before - wymaga ona podania argumentu określającego dla jakich metod i dla których klas ma być obsługiwane zdarzenie. Definiujemy więc tutaj zwany punkt przecięcia (ang. pointcut). Punkty przecięcia definiuje się przy pomocy tak zwanych wyrażeń przecięcia AspectJ. Podstawowe wyrażenie przecięcia ma następującą strukturę:

<expression>([<method_scope>] <return_type>
<qualified_class_name>.<method>(<parameters>))

gdzie:

  • expression - rodzaj zdarzenia, my będziemy używali zdarzenia określającego wykonanie metody - zdarzenie to zapisujemy przy pomocy słowa execution
  • method_scope - modyfikator widoczności metody, jeżeli podamy np. public, to zdarzenie będzie wywoływane podczas uruchamiania metod publicznych; modyfikator ten można pominąć
  • return_type - zwracany typ wartości przez metodę, dla której wykonania zdarzenia mają być generowane
  • qualified_class_name - nazwa klasy, dla której ma być generowane zdarzenie wraz z pakietem
  • method - nazwa metody, dla której mają być generowane zdarzenia
  • parameters - typu parametrów, które przyjmuje metoda; możemy zastosować zapis .. (dwie kropki) aby wskazać, że zdarzenie ma być generowane dla dowolnej ilości argumentów dowolnych typów

Przykład nr 1:

execution(* *(..))

Zdarzenie będzie wywołane dla każdej metody każdej klasy.

Przykład nr 2:

execution(public * com.kodilla.myClass.myMethod(java.lang.String))

Zdarzenie będzie wywołane dla każdej publicznej metody o nazwie myMethod w klasie myClass (bez względu na typ zwracanego wyniku), przyjmującej jeden argument typu String.

24.4. Wzorzec Obserwator

Wzorzec obserwator (ang. observer) jest wzorcem wykorzystywanym do rozgłaszania informacji. W sytuacji gdy mamy w programie obiekty, których stan jest obserwowany przez wiele innych obiektów, wzorzec ten idealnie się nadaje jako rozwiązanie.

We wzorcu obserwator poszczególni obserwatorzy "subskrybują" się na otrzymywanie informacji o zmianach w obserwowanych obiektach.

Przykładem może być na przykład forum dyskusyjne Kodilli - znajdują się na nim kanały tematyczne (np. help-java, help-java-tools itd), do których przypisani są kursanci. Kursanci otrzymują powiadomienia o nowych postach, które pojawią się w kanałach tematycznych.

Wzorzec obserwator jest użyteczny zwłaszcza wtedy, gdy ilość obserwatorów liczymy w tysiącach lub milionach (co ma miejsce w wielkich systemach, jak na przykład serwisy informacyjne CNN czy notowania giełd NASDAQ itd. - jednak w mniejszych rozwiązaniach również ma to bardzo istotne znaczenie).

Wyobraź sobie jak bardzo byłby obciążony serwer, gdyby każdy obserwator co chwila "odpytywał" obserwowany obiekt o to czy wystąpiły na nim jakieś zmiany - również wówczas, gdy żadnych zmian przez dłuższy czas nie ma. Powiadamianie o zmianach rozsyłane przez obserwowany obiekt do swoich "subskrybentów" zdecydowanie zmniejsza obciążenie serwera.

24.5. Wzorzec Adapter

Wzorzec adapter (ang. adapter) używany jest w sytuacji, gdy musimy skomunikować ze sobą dwa niekompatybilne obiekty (lub np. systemy). Niekompatybilne to znaczy takie, które przechowują dane w innych formatach lub oczekują interfejsów o innych metodach, zwracanych typach lub argumentach. Mówiąc krótko - niezgodne obiekty.

Jako przykład dla naszych rozważań posłuży przypadek, w którym stary system kadrowo-płacowy chcemy zintegrować z nowym systemem kadrowo-płacowym. Stary system przechowuje dane o pracownikach oraz informacje o ich wynagrodzeniu zasadniczym, natomiast nowy system kadrowo-płacowy będziemy wykorzystywali do obliczenia kosztów prowadzonej działalności gospodarczej, czyli do obliczenia łącznego kosztu wynagrodzenia pracowników oraz do realizacji wypłat.

Opis problemu

Załóżmy, że chcemy teraz te dwa systemy (stary i nowy) ze sobą "skomunikować" - chcemy, aby stary system przechowujący dane pracowników mógł skorzystać z nowych mechanizmów obliczania listy płac udostępnianych przez nowy system.

Widać jednak, że nie będzie to całkiem proste, ponieważ dane w jednym i w drugim systemie przechowywane są w całkowicie różnych formatach (w jednym jest to tablica Stringów, natomiast w drugim jest to kolekcja obiektów klasy Employee). Ponadto wynagrodzenia w jednym z systemów przechowywane są w polach typu double, a w drugim w polach typu BigDecimal.

Można ten problem rozwiązać na kilka sposobów - rekomendowanym podejściem jest jednak zastosowanie wzorca projektowego adapter.

Rozwiązanie problemu - adapter

Idea adaptera polega na wstawieniu pomiędzy dwa systemy dodatkowej klasy "tłumaczącej" dane i zachowania właściwe dla jednego systemu na dane i zachowania właściwe dla drugiego systemu. Klasę tę nazywamy właśnie adapterem - możesz skojarzyć to sobie z przejściówką, np. z USB-C na zwykłe USB starszego typu.

Klasa adaptera podzielona jest wewnętrznie na dwie części - jedną odpowiedzialną za komunikację z systemem, którego metodę chcemy wywołać (adaptee) oraz drugą odpowiedzialną za komunikację z systemem wywołującym (adapter).

W praktyce często stosuje się rozwiązanie polegające na utworzeniu klasy Adaptee, która "opakowuje" wywołanie metod systemu docelowego - osiąga się to poprzez zaimplementowanie w klasie Adaptee interfejsu wywoływanego systemu w taki sposób, że klasa ta wywołuje odpowiednie metody dostarczane przez wywoływany system (klasa Adaptee staje się wrapperem tych metod).

Następnie, klasę Adaptee rozszerza się (poprzez dziedziczenie przez klasę Adapter) i dodaje się funkcjonalności charakterystyczne dla interfejsu oczekiwanego przez system wywołujący. We wnętrzu metod adaptera następuje transformacja danych i przekazanie wywołań do "wewnętrznej klasy" Adaptee.

Typ BigDecimal a pieniądze

Typ BigDecimal, oprócz tego, że nadaje się do przechowywania dużych liczb, pozwala również na operacje na liczbach zmiennoprzecinkowych z dokładną precyzją. Zwykłe typy zmiennoprzecinkowe, jak na przykład double często wprowadzają zakłócenia (błędy) na dalszych miejscach po przecinku.

W przypadku operacji na pieniądzach jest to niedopuszczalne, wobec czego warto do przechowywania informacji o kwotach pieniężnych zastosować inne typy danych - na przykład BigDecimal, który bardzo dobrze sprawdza się w tej roli.
