32. Wysyłanie maili i korzystanie z Thymeleaf

Wyzwania:

  • nauczysz się tworzyć szablony maili,
  • stworzysz warunki w szablonach,
  • nauczysz się iterować w pętli z wykorzystaniem biblioteki Thymeleaf.

Ten moduł będzie poświęcony rozbudowie części aplikacji zajmującej się wysyłaniem maili. Nauczysz się jak zbudować podstawowy szablon mailingowy, aby Twoje maile wyglądały znacznie bardziej profesjonalnie. Dodatkowo poznasz moc biblioteki Thymeleaf, która wprowadza masę ciekawych możliwości do tworzenia szablonów, takich jak pętle, zmienne czy warunki.

32.1. Thymeleaf - co to jest?

Thymeleaf jest biblioteką wspierającą tworzenie widoków wewnątrz naszej aplikacji - szablonów maili i wszystkim co związane z relacją back-end -> front-end.

Thymeleaf pomoże nam w łatwy sposób okiełznać wstrzykiwanie danych do kodu HTML, co można wykorzystać m.in. podczas tworzenia maili lub widoków. Dowiesz się też jak zaimplementować proste instrukcje warunkowe, pętle oraz inne przydatne mechanizmy za pomocą tej biblioteki - miłej nauki!

Omówienie szablonu

Wewnątrz szablonu e-mail zawarliśmy tymczasowe teksty, które zapełnimy dynamicznymi wartościami przy pomocy biblioteki Thymeleaf.

PREVIEW_MESSAGE - miejsce, w którym wyświetlać się będzie część związana informacją czego dotyczy wiadomość e-mail. Często przed wyświetleniem całego maila możesz zauważyć skróconą wiadomość - to właśnie miejsce, w którym uzupełnimy taką informację:

WELCOME_MESSAGE - to nic innego jak wiadomość powitalna. Jesteśmy profesjonalistami, dlatego też chcemy mieć możliwość generowania bardziej spersonalizowanych wiadomości, jak np. “Drogi Jakubie!”, “Hej Aneta!”, “Sz. P. Elżbieto”.

MESSAGE_CONTENT - treść wiadomości, która będzie zawierać informacje o dodanej karcie w Trello.

BUTTON_NAME - nasz email będzie zawierać przycisk, który pozwoli na przekierowanie naszego użytkownika bezpośrednio do aplikacji.

GOODBYE_MESSAGE - wiadomość pożegnalna, czasami może istnieć potrzeba, aby, tak jak wypadku przywitania, stworzyć spersonalizowane pożegnanie.

COMPANY_DETAILS - stopka, czyli miejsce na dodatkowe informacje, chociażby o nazwie naszej firmy, adresie strony internetowej czy adresie email do kontaktu.

Konfiguracja silnika szablonów

Żeby nasz szablon współgrał z biblioteką Thymeleaf, należy przygotować dwie niezbędne rzeczy:

  1. Silnik szablonu - udostępniany przez bibliotekę Thymeleaf, który domyślnie jest już beanem. Pozwala on na interpretacje zmiennych na ustawione przez nas wartości w szablonie.
  2. Kontekst Thymeleaf - który odpowiada za przechowywanie i udostępnianie zmiennych bezpośrednio do szablonu.

Aby tego dokonać, utwórz nowy serwis w pakiecie com.crud.tasks.service o nazwie MailCreatorService. Następnie pobierz jako zależność TemplateEngine udostępniony w pakiecie org.thymeleaf.TemplateEngine.

Interfejsy funkcyjne w Javie

W związku z wprowadzeniem wyrażeń lambda w języku Java, pojawiło się pojęcie interfejsu funkcyjnego (ang. functional interface). Jest to specyficzny rodzaj interfejsu, posiadający tylko jedną metodę. Takie interfejsy doskonale nadają się do implementowania przez wyrażenia lambda (jak pamiętasz, zmienna przechowująca kod w postaci wyrażenia lambda jest właśnie typu takiego interfejsu).

Język Java w wersji 8 wyposażono w zestaw gotowych interfejsów funkcyjnych ogólnego przeznaczenia, takich jak na przykład:

@FunctionalInterface
public interface Predicate<T> {
  boolean test(T t);
}

lub

// The T is the input argument while R is the return result
@FunctionalInterface
public interface Function<T, R> {
  R apply(T t);
}

lub

@FunctionalInterface
public interface Consumer<T> {
  void accept(T t);
}

lub

@FunctionalInterface
public interface Supplier<T> {
  T get();
}

Interfejsów tych jest dużo więcej - ale nie musisz ich wszystkich znać. Zauważ, że wszystkie te interfejsy są opatrzone adnotacją @FunctionalInterface. Jest to adnotacja informacyjna, mówiąca iż przeznaczeniem interfejsu jest użycie go w programowaniu funkcyjnym (czyli np. w wyrażeniach lambda). Często będziesz spotykał takie interfejsy w metodach udostępnianych przez Stream.

32.4. Ostatni refactor REST API

Na koniec rozwoju naszej aplikacji, musimy pokusić się jeszcze o mały refactor, który spowoduje, że nasz system stanie się bardziej profesjonalny. Wykorzystamy pewien standard w projektowaniu REST API, z którym możesz spotkać się w pracy - Richardson Maturity Model.

Wcześniejsze podejście było przedstawione stricte "wizualizacyjnie", abyś mógł łatwiej zrozumieć te wszystkie endpointy i inne nowe rzeczy :)

Teraz, kiedy już masz pół roku doświadczenia za sobą, możemy wykonać ostatnie szlify w aplikacji i uczynić ją naprawdę czymś, czym można śmiało pochwalić się swoim znajomym lub rekruterowi :)

Richardson maturity model - Fundamentalne założenia

Jest to model opracowany przez Leonarda Richardsona, który opisuje w jaki sposób należy projektować systemy typu REST. Można nazwać ten model wzorową praktyką projektowania takich serwisów i warto go znać, ponieważ codziennie w pracy możesz natknąć się na zadanie związane z budowaniem nowego API czy utworzenie nowego endpointa w aplikacji.

Model Richardsona zakłada podział na poziomy 0, 1, 2 i 3 - im więcej z nich spełniamy, tym bardziej przyjazne jest nasze API. Musisz pamiętać, że Twoja aplikacja po zastosowaniu tego modelu nie zostanie nagle wyśmienitym REST’owym serwisem, który rozwiązuje wszystkie problemy świata! Jest to jedynie ważny standard, z którego warto korzystać, aby każdy deweloper mógł sprawnie odnaleźć się w projekcie. Opiszemy poszczególne poziomy poniżej.

Poziom 0

Poziom 0 mówi o tym, że Twoja aplikacja REST’owa nie jest nowym protokołem. Aplikacja jest oparta na protokole HTTP, a komunikacja z REST API odbywa się poprzez wysyłanie żądań i uzyskiwanie odpowiedzi - dokładnie tak, jak w przypadku pobierania zadań dostępnych w bazie danych.

Może Ci się to wydawać nieco mało zrozumiałe - w końcu jakiego innego protokołu mielibyśmy używać do komunikacji z API? Istnieją standardy takie jak SOAP, w których HTTP jest jedynie "kurierem" naszego zapytania do serwera, a całość zapytania jest opisywana w XML zawartym w ciele zapytania. Takie podejście nie jest dobrym sposobem na tworzenie REST API, więc nazywamy to poziomem 0.

Poziom 1

Mowa tu o deklarowaniu zasobów. Zastanów się - czym jest nasza aplikacja? Operujemy na zasobach, przetwarzamy je, warunkujemy, ale finalnie i tak udostępniamy określone zasoby - praktycznie wszystkie aplikacje służą do pobierania, modyfikowania i zapisywania zasobów. Poziom 1 mówi o nadawaniu odpowiedniej hierarchii zasobom naszej aplikacji.

Sednem poziomu 1 jest odpowiednie nazywanie adresów URL do zasobów aplikacji, tak aby było to zrozumiałe i czyste. Ten etap poznamy dokładnie w trakcie refaktoryzacji naszej aplikacji, po omówieniu poziomu 3.

Poziom 2

Ten poziom zastosowaliśmy w naszym REST API dawno temu - mówi on o tym, aby korzystać z dostępnych metod protokołu HTTP takich jak GET, POST, PUT, DELETE, aby aplikacja była schematyczna i zgodna z dobrymi praktykami - przecież nikt nie wymyślał tego typu metod po to, aby z nich nie korzystać. Metody HTTP służą do celów programistycznych, abyśmy wiedzieli do czego służy dany endpoint, ale są również przydatne w tworzeniu dokumentacji.

Jak wiesz, nasze API korzysta z tych metod, więc nie jest to nic nowego dla Ciebie!

Poziom 3

Jest to poziom, którego zastosowanie skupia się na wskazywaniu drogi do innych endpointów aplikacji. Załóżmy przypadek, w którym nasza aplikacja umożliwia pobieranie produktów, które następnie można zakupić również przez nasze API, wykorzystując numer identyfikacyjny i ilość do zamówienia.
